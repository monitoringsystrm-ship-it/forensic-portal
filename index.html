1 Introduction

1.1 Purpose

The purpose of this document is to define the software requirements for the Forensic Monitoring System for CI/CD pipelines and related infrastructure. It serves as a contractual basis between stakeholders, developers, and testers, providing a stable description of the system’s behavior, interfaces, and constraints, independent of specific implementation details.

1.2 Scope

This document covers functional and non-functional requirements for the Forensic Monitoring System. In scope: (1) Forensic-Ready CI/CD Agents—Jenkins and GitHub integration, build evidence capture, Docker context, command-sequence ML (benign/suspicious/malicious) with explainability; (2) SBOM and Malware—pipeline-stage SBOM (CycloneDX-style), Pipeline Merkle tree, verification service (pipeline integrity, CVE-affected builds, artifact provenance), Jenkins shared library (sbomCapture, publishPipelineSBOM), malware hash lookup; (3) Automated Incident Analysis—GitHub commit and dependency history ingestion, Jenkins build metrics, threat intel (CVE, npm/GitHub advisories, compromised packages), commit and dependency anomaly reports, incident timeline and trend analysis; (4) Chain of Custody and Integration—single flow from GitHub push through capture, SBOM, ML alerts, and block/alert decision; incident timeline visualization, pipeline integrity UI, forensic query by CVE and artifact hash, Merkle tree visualization. It addresses the React front-end, Flask/Python backend, and Jenkins/GitHub/Docker integrations; it excludes detailed deployment automation and third-party SOC processes that consume the system’s outputs.

1.5 Overview

The Forensic Monitoring System is a web-based platform targeting Jenkins + GitHub + Node.js/React + Docker. It integrates with CI/CD to: (1) ingest GitHub webhooks (push, PR) and Jenkins build payloads (stages, console output, env, Docker context); (2) run a command-sequence ML classifier and store timestamped evidence and commit→build mapping; (3) capture pipeline-stage SBOMs via an orchestrator and build a Pipeline Merkle tree, with verification and forensic query (CVE, artifact provenance); (4) ingest commits, dependency history, Jenkins build metrics, and threat intel to produce commit/dependency anomaly reports and an incident timeline; (5) expose an integrated flow (GitHub → capture → SBOM → ML → block/alert) and UI views for timeline, pipeline integrity, forensic query, and Merkle visualization. It comprises a React front-end and a Flask/Python backend with REST APIs for training, inference, evidence, SBOM capture/finalize, verification, and reporting. Primary users include security engineers, DevOps, and compliance personnel.

2 Overall Descriptions

This section provides a high-level description of the system and its operating context, intended to remain relatively stable across multiple releases.

2.1 Product Perspective

The Forensic Monitoring System is an independent web application that can be deployed alongside existing CI/CD infrastructure such as Jenkins or GitHub Actions. Unlike traditional monitoring dashboards that focus primarily on performance or availability metrics, this system emphasizes forensic readiness, evidentiary integrity, and investigation support. Compared to generic SIEM or log aggregation tools, it adds specialized ML-driven anomaly detection over build logs and network data, chain-of-custody tracking for evidence, and standards-aligned reporting such as ISO/IEC 27037 and NIST SP 800-86.

2.1.1 System Interfaces

The system is implemented as a web server exposing REST APIs and a browser-based single-page application. It interfaces with:

  • Inbound: GitHub webhooks (push, pull_request) and Jenkins build POSTs (build metadata, console output, env, Docker context); optional CI/CD agents posting evidence.
  • File system: Local datasets (HDFS/Hadoop logs, NSL-KDD, UNSW-NB15, malware CSVs, attack/breach CSVs); pipeline SBOM store and evidence/CICD state under data/.
  • Network: HTTP/HTTPS for REST API and UI; outbound optional (e.g. build status to GitHub).

2.1.2 User Interfaces

The primary user interface is a responsive web dashboard with the following areas:

  • Forensic-Ready CI/CD Agents: Agent status, evidence list, config (capture options); HDFS/Hadoop anomaly summaries; command sequence train/infer and command sequence alerts (build_id, command, confidence, risk_level) with link to evidence.
  • Anomaly Detection: Commit/dependency/pipeline-tampering detection and training; Incident Analysis ingest (GitHub commits, dependency history, Jenkins patterns, threat intel), analyze, and latest report; incident timeline and commit/dependency anomaly reports (reasons, recommendation, typosquatting, BLOCK_INSTALL, similar_package).
  • Integrity Verification: SBOM hash, SHA-256 verify, Merkle root from hashes, HMAC/Ed25519 sign/verify; pipeline SBOM capture/finalize, list pipelines, view pipeline stages and per-stage SBOM; verify pipeline; forensic query by CVE (affected builds) and by artifact hash (provenance chain); Merkle tree visualization (e.g. D3.js); Microsoft Malware classifier and malware hash lookup.
  • Forensic Reporting Portal: Evidence visualization, chain-of-custody, integrity proofs, attack/breach ML models, report generator; incident timeline and pipeline integrity status in reporting context.
  • Integrated Flow: Single flow status and decision (GitHub push → capture → SBOM → ML → block/alert); pipeline view with full flow and incident timeline.

The interface is optimized for desktop but remains usable on tablets and mobile devices through responsive layout.

2.1.3 Hardware Interfaces

The system does not require specialized hardware beyond commodity servers or developer workstations. It operates on standard processor architectures with sufficient memory and disk capacity to store datasets and generated reports.

2.1.4 Software Interfaces

The system interfaces with the following software components:

  • Modern web browsers supporting ES6 JavaScript and the Fetch API.
  • Python runtime and Flask; ML libraries (e.g. scikit-learn, numpy, pandas) for training, inference, and visualization.
  • Operating-system file system for datasets, models, reports, and persisted state (e.g. data/cicd_state.json, data/pipeline_sbom_store.json, data/incident_analysis_state.json).
  • Jenkins via shared library (Groovy) calling backend REST APIs; GitHub via webhook delivery to backend; Docker context provided in Jenkins build payloads.

2.1.5 Communication Interfaces

Client and server communication uses HTTP or HTTPS with JSON payloads over a REST-style API. Within a deployment, the system may also communicate with CI/CD agents or other internal services using HTTP-based endpoints or file drops, depending on site integration patterns.

2.1.6 Memory Constraints

The system is intended to run on machines with at least 8 GB of RAM for moderate-size datasets. Training jobs on larger datasets may benefit from 16 GB or more. At runtime the front-end has no special memory constraints beyond typical browser limits, while the backend should stay within the available memory when loading models and computing reports.

2.1.7 Operations

Normal operations include:

  • Starting and stopping the backend API service.
  • Running the front-end web server or serving static assets.
  • Triggering model training and monitoring progress from the UI.
  • Running anomaly detection jobs, evidence collection, and malware classification.
  • Generating and exporting forensic reports.

Special operations include system upgrades, dataset replacement or refresh, model re-training on new corpora, and archival of legacy evidence or report data.

2.1.8 Site Adaptation Requirements

Each installation may adapt the system by configuring:

  • File paths to locally stored logs and datasets.
  • CI/CD agent configuration for evidence capture frequency and scope.
  • Integration endpoints or secure storage locations for generated reports.

Deployment environments should provide appropriate network access policies, TLS termination, and storage quotas consistent with local security and compliance requirements.

2.2 Product Functions

Major functions of the Forensic Monitoring System include:

  • Ingestion of GitHub webhooks (push, PR) and Jenkins build payloads (stages, console, env, Docker); storage of evidence with build UUID and timestamps; commit→build mapping.
  • Command-sequence ML: tokenize console into command sequences; classify benign/suspicious/malicious with confidence, risk_level, and explainability; persist command alerts and link to evidence.
  • Pipeline-stage SBOM: capture_stage_context (environment, tools, inputs, outputs, dependencies); extended CycloneDX-style schema; Pipeline Merkle tree (add_stage, finalize_pipeline, verify_stage) with proof chain per pipeline_id.
  • Forensic verification: verify_pipeline_integrity(pipeline_id); query_vulnerable_builds(cve_id); trace_artifact_provenance(artifact_hash).
  • Jenkins shared library: sbomCapture('stageName') { ... } and publishPipelineSBOM() for CI integration.
  • Incident analysis: ingest GitHub commits, dependency history, Jenkins build metrics, threat intel (CVE, npm/GitHub advisories, compromised packages); analyze with Isolation Forest and rule-based features; output commit anomaly report, dependency risk assessment, incident timeline, trend analysis.
  • Integrity and malware: SBOM hash, Merkle root, HMAC/Ed25519; Microsoft Malware classifier; malware hash lookup (e.g. known families); pipeline integrity and forensic query UIs.
  • Integrated flow: single flow GitHub → Component 1 (capture + command ML) → Component 2 (SBOM + malware) → Component 3 (commit/dep analysis) → Component 4 (block/alert); decision API for deployment gating.
  • Chain-of-custody, evidence correlation, attack/breach classifiers, and standards-aligned forensic report generation.

2.3 User Characteristics

Typical users of the system include:

  • Security engineers with experience in incident response and several years of working with logs and monitoring tools.
  • DevOps and SRE practitioners familiar with CI/CD tools, build pipelines, and deployment workflows.
  • Compliance officers and auditors who may have limited technical depth but need consistent, readable reports and evidence summaries.
  • Data scientists or ML engineers who may operate and tune the underlying models.

Users are assumed to be comfortable with web interfaces and basic security concepts. No programming background is strictly required to operate the system.

2.4 Constraints

  • The system must use HTTP or HTTPS and JSON for its public API to remain compatible with browsers and common automation tools.
  • The implementation is constrained to the existing technology stack of React on the client and Flask with Python ML libraries on the server.
  • Data and models are stored on the host file system and no external database is assumed in the baseline design.
  • Use of public datasets for malware or intrusion detection may be restricted by licensing or organizational policy.
  • Security and privacy policies may limit which environment variables, logs, or secrets can be collected or retained as evidence.

2.5 Assumptions and Dependencies

  • The system will run on modern operating systems with a supported Python runtime.
  • Users will access the system through up to date browsers that support standard web APIs.
  • Required training datasets are present and accessible at configured file paths on the backend host.
  • Time synchronization is available so that timestamps across evidence items and logs are comparable.
  • Future versions may depend on direct integration with CI/CD tools via APIs or webhooks.

2.6 Apportioning of Requirements

Initial releases prioritize:

  • Core anomaly detection and CI/CD evidence collection.
  • Cryptographic hashing and basic chain-of-custody views.
  • Baseline reporting capabilities for internal use.

Subsequent releases may extend:

  • Advanced ML visualizations and custom model configuration.
  • Deeper CI/CD platform integrations and alerting.
  • Role based access control, multi-tenant support, and long-term archival workflows.

3 Specific Requirements

3.1 External Interface Requirements

3.1.1 User Interfaces

The system shall provide a browser-based UI with the following:

  • Main navigation: tabs for CI/CD Agents, Anomaly Detection, Integrity Verification, Reporting Portal, and Integrated Flow.
  • CI/CD Agents: agent status, evidence list, config; HDFS/Hadoop anomaly summaries; command sequence train/infer and command sequence alerts (build_id, command, confidence, risk_level) linked to evidence.
  • Anomaly Detection: commit/dependency/pipeline-tampering detection and training; Incident Analysis ingest (commits, dependency history, Jenkins patterns, threat intel), analyze, latest; incident timeline and commit/dependency anomaly reports (reasons, recommendation, BLOCK_INSTALL, similar_package).
  • Integrity Verification: SBOM hash, SHA-256 verify, Merkle root; pipeline SBOM capture/finalize, list pipelines, pipeline detail (stages, per-stage SBOM); verify pipeline; forensic query by CVE and by artifact hash; Merkle tree visualization; Microsoft Malware and hash lookup.
  • Reporting Portal: evidence visualization, chain-of-custody, integrity proofs, attack/breach models, report generator; incident timeline and pipeline integrity in reporting context.
  • Integrated Flow: flow status and decision (block/alert/allow) for pipeline view with full flow and incident timeline.

All screens shall be usable on displays down to common tablet and phone widths through responsive layout.

3.1.2 Hardware Interfaces

The software shall not require direct control of specialized devices. It shall rely on:

  • Standard network interfaces configured by the host operating system.
  • Local storage volumes where datasets and model artifacts are stored.

3.1.3 Software Interfaces

The system shall interface with:

  • The host operating system file system for reading datasets and writing trained models and reports.
  • Python libraries for data loading, model training, visualization, hashing, and cryptographic operations.
  • Web browsers that invoke REST endpoints exposed by the backend.

3.1.4 Communication Interfaces

The backend shall expose REST APIs using HTTP or HTTPS on a configurable port. All application level messages between front end and backend shall use UTF-8 encoded JSON. For secure deployments, HTTPS termination shall be provided either directly by the application server or via a reverse proxy.

3.2 Classes/Objects

The following conceptual classes organize the detailed requirements. Each is categorized as essential, desirable, or optional.

  • EvidenceItem (essential): represents a unit of collected evidence with fields such as identifier, type, timestamp, description, metadata, and integrity checksum.
  • AnomalyRecord (essential): represents a detected anomaly with fields for model, category, confidence, severity, and supporting details.
  • CicdAgent (essential): represents a monitoring agent with identifier, type, status, metrics, and configuration.
  • IntegrityProof (essential): represents a cryptographic proof including hash, algorithm, optional signatures, Merkle roots, and timestamps.
  • CustodyLog (essential): represents a chain-of-custody entry including evidence ID, custodian, location, status, history, and checksum.
  • MlModelState (desirable): represents the training and evaluation status of an ML model including metrics and available reports.
  • ForensicReport (essential): represents a generated report with identifier, standard, format, generated timestamp, summary, sections, and embedded data.
  • UserSession or AccessContext (optional): represents user identity and roles if authentication and authorization are introduced in later versions.
  • PipelineSBOM / StageSBOM (essential): pipeline-stage SBOM with metadata.pipelineStage, buildEnvironment, inputArtifacts, outputArtifacts, components; CycloneDX-style.
  • PipelineMerkleProof (essential): root_hash, proof_chain, leaf_hashes, stages; supports verify_stage(stage_index, sbom, root, proof).
  • CommandAlert (essential): build_id, command, confidence, risk_level, why (explainability); linked to evidence.
  • IncidentAnalysisReport (essential): commit_anomalies, dependency_risks, build_anomalies, timeline, trends; anomaly_type, reasons, recommendation, BLOCK_INSTALL, similar_package.

3.3 Performance Requirements

  • The UI shall display core dashboard views within a reasonable time under normal network conditions for moderate data volumes.
  • The system shall support displaying several thousand evidence items without becoming unusable for filtering and basic navigation.
  • Batch anomaly detection and model training jobs may run for minutes depending on dataset size, and the system shall expose completion indicators.
  • Integrity checks such as SHA-256 computation and Merkle root calculation shall complete interactively for typical text or JSON payloads.

3.4 Design Constraints

  • The client shall be implemented as a single-page application using React.
  • The backend shall be implemented in Python using Flask for HTTP APIs.
  • ML functionality shall use widely available open source libraries to ease deployment and reproducibility.
  • The system shall maintain a clear separation between UI, API, and ML logic to support future changes and testing.

3.5 Software System Attributes

3.5.1 Reliability

The system shall handle invalid inputs and missing datasets gracefully, returning informative error messages rather than failing silently. Evidence and anomaly data persisted to disk shall be written in a way that reduces the risk of partial updates.

3.5.2 Availability

The system is intended for continuous operation during working hours and may be left running for extended periods. Administrators shall be able to restart the backend service without losing previously stored models, evidence, or reports.

3.5.3 Security

The system shall avoid exposing sensitive environment details or raw secrets in the UI or logs. When deployed in production, communications should be secured via HTTPS, and access to datasets and report directories shall be limited to authorized users. Future enhancements may add authentication, authorization, and auditing for user actions.

3.5.4 Maintainability

The system shall be modular, with separate components for CI/CD agents, anomaly detection, integrity verification, and reporting. Configuration values such as dataset paths and ports shall be externalized where possible. Code organization should facilitate unit and integration testing of key functions.

3.6 Other Requirements

  • The UI text and reports shall be written in clear, concise English.
  • Generated reports shall be exportable in common formats such as JSON and shall provide structure to support future PDF or XML rendering.
  • The system shall avoid hard coding site specific paths where practical to ease relocation across environments.

5 Implemented Features and Capabilities

This section summarizes the implemented features aligned with the Forensic-Ready CI/CD Security Framework (see NEW_FEATURES_AND_IMPLEMENTATION.md). All items below are implemented in the current system.

5.1 Component 1 – Forensic-Ready CI/CD Agents

  • Tech stack: Jenkins + Node.js/React + Docker (in addition to optional HDFS/Hadoop legacy agents).
  • Inputs – GitHub: Webhooks for push, PR; commit SHA, author, timestamp; ingest and commit→build mapping.
  • Inputs – Jenkins: Jenkinsfile/pipeline stages (build/test/dockerize); build_id, stage_name, stage_index, commit_sha, branch, tags; Node/npm/yarn context.
  • Inputs – Build env: Redacted env vars, Docker daemon context; package.json/lock snapshots where applicable.
  • Inputs – Runtime: Jenkins console output, tokenized command sequences, Docker build commands, npm install/build.
  • Inputs – Docker: Dockerfile, build args, image metadata, tags (sent in Jenkins build payload).
  • Outputs – Evidence: Timestamped build logs (JSON), env dumps (redacted), command traces; evidence list with types (build-logs, env-vars, docker-build, command-alert, github-event).
  • Outputs – Docker: Image/tags and build context metadata in evidence.
  • Outputs – GitHub: Commit→build mapping (SHA→build_id); API for mappings.
  • Chain of custody: SHA-256 hashes, build UUID, timestamps; signing (HMAC, Ed25519) available.
  • ML – Command sequence: Tokenized command sequences; classifier (benign/suspicious/malicious); confidence, risk_level; per-command anomaly score; alert JSON with build_id, command, confidence, risk_level, explainability (why).

5.2 Component 2 – SBOM and Malware Detection

  • Pipeline-stage SBOM: Per-stage capture of env, tools, inputs, outputs, deps; extended CycloneDX-style schema with pipelineStage, buildEnvironment, inputArtifacts, outputArtifacts, components.
  • Orchestrator: PipelineSBOMOrchestrator – capture_stage_context(), capture_environment(), capture_tools_used(), capture_inputs/outputs(), generate_dependency_sbom().
  • Merkle integration: Pipeline Merkle tree – add_stage(stage_sbom), finalize_pipeline() → root + proof chain, verify_stage(); leaf = stage_name, index, sbom_hash, timestamp.
  • CI integration: Jenkins shared library – sbomCapture('stageName') { ... }, publishPipelineSBOM().
  • Verification service: verify_pipeline_integrity(pipeline_id), query_vulnerable_builds(cve_id), trace_artifact_provenance(artifact_hash).
  • Malware: Microsoft Malware classifier (PE); SBOM hash lookup (e.g. known malware families); Node/npm context where applicable.

5.3 Component 3 – Automated Forensic Incident Analysis

  • GitHub commits: Ingest full commit history; metadata, diffs; contributor behavior and time patterns.
  • Contributor behavior: Commit frequency, time patterns; features in commit anomaly model.
  • Dependency history: package.json/lockfile changes, version bumps, new deps; ingest and analyze.
  • Jenkins patterns: Build frequency, duration, success/failure, resource use, Docker image size; ingested and used in trend analysis.
  • Threat intel: CVE, npm advisories, GitHub advisories, compromised package lists; ingest and use in dependency/build risk scoring.
  • ML – Commit anomaly: Isolation Forest + rule-based features; anomaly_type, reasons, recommendation (MANUAL_REVIEW, ELEVATED_MONITORING, ALLOW).
  • ML – Dependency anomaly: Typosquatting, Levenshtein, Isolation Forest; BLOCK_INSTALL, similar_package, detection_method.
  • Outputs: Commit anomaly report JSON; dependency risk assessment JSON; incident timeline (commit→build→deploy); trend analysis.

5.4 Component 4 – Chain of Custody and Integration

  • Integration flow: Single flow: GitHub push → Component 1 (capture + command ML) → Component 2 (SBOM + malware) → Component 3 (commit/dep analysis) → Component 4 (block/alert).
  • UI – Incident timeline: Timeline visualization with commit → build → deployment and anomaly/alert annotations.
  • UI – Pipeline integrity: Pipeline list, pipeline detail, per-stage SBOM summary, Merkle root and proof chain; “Verify pipeline” action.
  • UI – Forensic query: Query by CVE (affected builds); query by artifact hash (provenance chain + verification status).
  • UI – Merkle tree visualization: Tree of pipeline stages and hashes (e.g. D3.js).
  • UI – Command sequence alerts: List alerts (build_id, command, confidence, risk_level) under CI/CD Agents; link to evidence.
  • UI – Commit and dependency anomaly reports: Display reasons, recommendation, typosquatting, BLOCK_INSTALL, similar_package.
